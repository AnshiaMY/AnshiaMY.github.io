Embedded Car Dashboard

A Raspberry Pi Pico-based dashboard system that uses Python input/output logic, a fabricated PCB, push-button inputs, LED indicators, a potentiometer-controlled OLED fuel display, and temperature-based warning behaviour to simulate a vehicle instrument panel.

Project Overview

From Pins to Panels.

This project involved building a working embedded car dashboard from PCB fabrication to final programming. The system combined a Raspberry Pi Pico, custom circuit board, soldered components, push-button inputs, LED indicators, an OLED fuel display, and sensor-based warning logic into one physical dashboard prototype.

Build Stage 01

PCB Fabrication

The project began with designing a custom PCB layout for the dashboard control system. This stage focused on planning how each component would connect electrically while still fitting cleanly within the physical board layout.

After the layout was prepared, the design was transferred onto a copper board, etched to form the circuit traces, drilled for connection points, and cleaned for assembly. This gave me hands-on experience moving from a circuit layout to a physical board that could support the final embedded system.

Build Stage 02

Circuit Assembly

Once the PCB was fabricated, the components were soldered and connected to form the dashboard control circuit. This included the Raspberry Pi Pico, push buttons, LEDs, resistors, potentiometer, OLED display connections, and ribbon-cable wiring.

This stage required careful attention to component orientation, solder quality, and connection reliability. Because the dashboard had several indicators and inputs, one weak joint or misplaced connection could affect the entire input/output chain.

Build Stage 03

Embedded Control

The Raspberry Pi Pico was programmed in Python to handle the dashboard’s input/output logic. Push-button inputs were mapped to specific LED indicators, while the potentiometer provided an analog input for the OLED fuel display.

The program also handled sensor-based behaviour by triggering the temperature warning indicator when heat was detected. This connected the physical inputs, GPIO pin assignments, and dashboard outputs into one controlled embedded system.

Build Stage 04

System Testing

The final prototype was tested one signal path at a time. I checked each button, LED, OLED response, potentiometer reading, and temperature-triggered warning to confirm that the software behaviour matched the physical dashboard output.

When a feature did not work as expected, I traced the issue through the code, GPIO mapping, PCB trace, solder joint, ribbon cable, and component connection. This made debugging more systematic and helped separate software issues from hardware connection problems.

Troubleshooting

Finding the Faulty Link.

The main debugging challenge was verifying that software behaviour matched the physical circuit. A visible output issue could come from the Python logic, Pico pin assignment, PCB trace, solder joint, ribbon-cable connection, LED orientation, or input component.

Debug File 01

Verifying Input/Output Mapping

Symptom

Some indicators needed verification because each physical input had to activate the correct dashboard symbol. A mismatch between the button, Pico pin, or LED connection could make the panel display the wrong vehicle signal.

Diagnosis

The system had multiple layers between input and output: Python variables, Pico GPIO pins, PCB connections, ribbon cable paths, and LED placements on the faceplate. Debugging required checking the complete signal path instead of assuming the issue was only in the code.

Adjustment

I tested each button and output individually, then confirmed the complete mapping from input component to Pico pin to dashboard indicator. This made it easier to isolate pin-assignment errors, wiring swaps, or physical connection issues.

Debug File 02

Improving Wiring and Soldering Reliability

Symptom

With many LEDs, resistors, ribbon cables, and soldered connections, some parts of the prototype were sensitive to movement during handling and testing. A weak connection could cause an indicator to flicker, fail, or behave inconsistently.

Diagnosis

The issue was not only electrical; it was also mechanical. The dashboard panel, PCB, and wiring had to remain connected while being handled as one physical prototype, so strain on wires or solder joints affected reliability.

Adjustment

My partner and I retested the wiring, checked solder joints, reinforced weak connections, and verified continuity where needed. This improved the stability of the prototype during the final functionality demonstration.

Debug File 03

Calibrating Analog and Sensor-Based Inputs

Symptom

The OLED fuel display and temperature warning needed to respond clearly to input changes. If the potentiometer reading or temperature threshold was not interpreted correctly, the dashboard feedback felt less realistic.

Diagnosis

These features depended on converting physical input behaviour into software-driven output. The potentiometer value had to be translated into a readable fuel level, while the temperature-sensitive input needed a threshold that triggered the warning light reliably during the demonstration.

Adjustment

I tested the input response and refined the logic so the OLED display and temperature indicator behaved consistently. This helped the dashboard simulate more realistic vehicle feedback instead of only demonstrating simple on/off LEDs.

Final Takeaway

Final Reflection.

What This Project Taught Me

This project taught me how input/output mapping works in a real embedded system. I became more confident understanding how components fit together electrically and physically, from Pico GPIO pins and PCB traces to soldered connections, ribbon cables, LEDs, buttons, the potentiometer, and the OLED display. The biggest technical lesson was learning that the code, circuit layout, and physical assembly all have to match for each dashboard signal to reach the correct output.

This project also taught me how to work under a tight timeline. Since the prototype had to be completed in about 20 days, my partner and I had to make practical design decisions, divide the work clearly, and test each section as we built it instead of waiting until the end. That time constraint pushed me to be more organized, troubleshoot efficiently, and focus on building a reliable working system rather than overcomplicating the design.

Future Improvements

Next Iteration Notes

  • Design a cleaner enclosure so the PCB, ribbon cables, and dashboard faceplate are protected during handling.
  • Improve cable management with stronger connectors, strain relief, and cleaner routing between the PCB and display panel.
  • Add more realistic dashboard behaviour, such as blinking turn signals, startup checks, warning-priority logic, or multiple alert states.
  • Refine the OLED fuel display with smoother scaling, clearer graphics, and a more vehicle-style gauge layout.
  • Expand the system with additional sensor inputs to simulate more vehicle conditions beyond button-controlled indicators.
