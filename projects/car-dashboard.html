Project Overview

Building an embedded dashboard prototype.

This project focused on designing and building a physical car dashboard prototype that could simulate common vehicle indicators through a custom embedded system. Instead of only creating a visual mockup, the dashboard combined a fabricated PCB, Raspberry Pi Pico control, push-button inputs, LED outputs, an OLED fuel display, and a temperature-sensitive warning input.

This was a partner project with shared wiring, assembly, and testing responsibilities. My main contributions were the Python control logic and PCB fabrication, while my partner and I worked together to connect the physical dashboard panel, ribbon cables, LEDs, buttons, and microcontroller into one working prototype.

Build Stage 01

Dashboard Interface Design

The dashboard was designed to resemble a vehicle-style instrument panel with recognizable warning symbols, turn indicators, gauge graphics, and a central status area. The goal was to create a physical interface that made the electronics easier to understand because each light represented a familiar driver-facing signal.

The front panel included indicators such as seatbelt, battery, oil, engine, vent, temperature, and left/right turn signals. This gave the project a clear user-facing purpose: button presses and sensor changes would not just turn on random LEDs, but simulate how a real dashboard communicates vehicle status to a driver.

Build Stage 02

PCB & Pico Integration

The control side of the project was built around a Raspberry Pi Pico programmed in Python. The Pico acted as the main controller, reading input signals and driving the dashboard outputs. A custom fabricated PCB was used to organize the microcontroller, buttons, connectors, and supporting components into a cleaner embedded system.

Ribbon cables connected the PCB to the dashboard faceplate so the control board could sit behind or beside the display panel while still driving the LEDs and other outputs. This created a more realistic embedded layout than a simple breadboard circuit because the control electronics and physical interface had to be connected through structured wiring.

Build Stage 03

Inputs, Sensors & Indicators

The dashboard used push-button inputs to control different LED indicators, allowing each dashboard warning or signal light to be tested individually. The Python code mapped input states to output behaviour so that button presses could activate the corresponding symbols on the physical display.

The prototype also included a potentiometer-controlled OLED fuel display. Rotating the potentiometer changed the fuel value shown on the display, which simulated an analog dashboard input being converted into a readable digital output. A temperature-sensitive input was also used so heat could trigger the thermometer warning light, adding a sensor-based interaction beyond simple button control.

Build Stage 04

Testing & Final Assembly

Once the PCB, Pico, wiring, LEDs, OLED display, potentiometer, and dashboard panel were connected, the project required repeated testing to confirm that each input triggered the correct output. This included checking button-to-LED mapping, fuel display response, temperature-warning behaviour, and the reliability of the ribbon cable connections.

Final assembly involved both electrical and physical constraints. The dashboard had to remain readable from the front while the wiring and components stayed secure on the back. Testing helped identify connection issues, alignment problems, and places where the wiring needed to be reinforced for a more reliable demonstration.

Troubleshooting

Debugging the embedded dashboard.

The main debugging challenges came from connecting software logic to a physical circuit. Even when the Python code was correct, the final behaviour depended on reliable soldering, correct pin mapping, stable wiring, and proper communication between the PCB and dashboard faceplate.

Debug File 01

Mapping inputs and outputs correctly

Symptom

During testing, each button and indicator needed to match the correct dashboard symbol. If a button controlled the wrong LED or an indicator did not respond, the dashboard no longer behaved like a clear vehicle-style interface.

Diagnosis

The issue could come from either the Python pin assignments or the physical wiring. Since the project used a PCB, ribbon cables, and multiple LEDs, debugging required checking both the software mapping and the electrical path between the Pico and the dashboard panel.

Adjustment

I tested each input and output individually, then verified the full mapping between buttons, Pico pins, ribbon cable connections, and dashboard LEDs. This made it easier to isolate whether an issue came from code, wiring, or a physical connection.

Debug File 02

Improving wiring and soldering reliability

Symptom

Some connections were sensitive to movement because the project used many physical wires, resistors, LEDs, ribbon cable connections, and solder joints. A loose or weak connection could cause an indicator to flicker or fail during testing.

Diagnosis

The physical build introduced constraints that would not appear in code alone. The dashboard panel had to hold several LEDs and wires while the PCB remained connected through ribbon cables, so mechanical strain and connection quality affected electrical reliability.

Adjustment

My partner and I reinforced and retested the wiring, checked solder joints, and verified continuity where needed. This helped make the dashboard more reliable during the final demonstration and reduced issues caused by loose physical connections.

Debug File 03

Testing the OLED display and temperature input

Symptom

The fuel display and temperature warning needed to respond clearly to changing inputs. If the potentiometer value did not update smoothly or the temperature input was not sensitive enough, the dashboard feedback became less convincing.

Diagnosis

These features depended on converting physical input changes into software-driven dashboard outputs. The potentiometer had to be read and translated into a fuel display value, while the temperature-sensitive component had to trigger the correct warning indicator when heat was applied.

Adjustment

I tested the input response and adjusted the logic so the display and warning indicator behaved consistently during the demo. This helped connect the dashboard prototype to more realistic vehicle-style feedback instead of only using simple on/off LED controls.

Final Takeaway

Final Reflection.

What this project taught me.

This project helped me understand how embedded systems connect software, electronics, and physical interface design. Writing the Python logic was only one part of the work; the final dashboard also depended on PCB fabrication, soldering, wiring, component placement, and testing each input-output connection.

I also learned how important pin mapping and systematic debugging are when software is controlling real hardware. If an LED did not respond correctly, the issue could be in the code, the PCB, the ribbon cable, the solder joint, or the physical LED connection. This pushed me to test one signal path at a time instead of assuming the problem was only in the program.

Personally, this project gave me more confidence working with physical prototypes that are not perfectly clean or predictable. The dashboard required patience, hands-on troubleshooting, and teamwork because the final result depended on both the circuit and the physical assembly behaving reliably together.

Future Improvements

Next iteration notes

  • Design a cleaner enclosure or mounting system so the PCB, ribbon cables, and dashboard faceplate are protected and easier to handle.
  • Improve wiring organization by using more structured connectors, strain relief, and cleaner cable routing between the PCB and dashboard panel.
  • Add more realistic dashboard behaviour, such as blinking turn signals, startup checks, warning priority logic, or different alert states.
  • Refine the OLED fuel display with smoother value scaling, clearer graphics, or more vehicle-style visual feedback.
  • Expand the prototype with additional sensors or inputs to simulate more vehicle conditions beyond button-controlled warning indicators.
