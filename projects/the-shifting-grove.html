The Shifting Grove

A Python and Pygame fantasy maze game where players explore a procedurally generated enchanted grove, collect shards, complete three integrated mini-games, and outsmart an AI-controlled fox guardian through pathfinding and reward-based progression.

Project Overview

Building a World Through Code.

The Shifting Grove began as a personal software project inspired by my interest in creating interactive and imaginative experiences. I wanted to move beyond smaller programming exercises by building a complete game with structured logic, responsive gameplay, and a polished user experience.

The final concept is a fantasy maze game where players explore an enchanted grove, collect Grove Shards, complete mini-game challenges, and race to escape before the grove shifts. The project allowed me to turn algorithms and software concepts into a playable experience while strengthening my programming, debugging, and software-architecture skills.

Development Stage 01

Game Architecture

The game was built using a modular, object-oriented architecture so each major system could be developed, tested, and debugged independently. Instead of placing all logic inside one large script, I separated the project into focused files and classes for the player, fox, maze generation, shards, UI, asset loading, mini-games, settings, and the main game controller.

A major part of the architecture was explicit state management. The game includes a start menu, instructions, maze gameplay, mini-game transitions, reward screens, win and loss states, and return-to-maze logic. Each state needed defined entry conditions, input handling, update behaviour, drawing logic, and exit transitions so one screen or mini-game would not interfere with another.

This structure became especially important when the project expanded from one maze into a complete game with three integrated mini-games. The architecture had to preserve player progress, apply rewards, reset temporary mini-game variables, and return to the correct maze state after each challenge.

Development Stage 02

Maze & Player Systems

The main gameplay is built around procedural maze generation. A randomized depth-first search approach creates a connected grid by carving paths between cells, allowing each round to use a new navigable layout instead of a manually fixed map.

The generated maze data connects directly to the rest of the game. Wall tiles, walkable paths, collision boundaries, shard locations, mini-game trigger areas, fox routes, and the final portal objective all depend on the generated grid.

The player system handles movement, wall collision, collectible logic, portal activation, and transitions into mini-games. This required the movement system, maze grid, collision logic, and overall state flow to remain synchronized throughout each round.

Development Stage 03

Fox AI & Game Logic

The fox AI was one of the project’s main technical systems. Rather than relying only on random movement or a fixed chase sequence, the fox uses breadth-first search pathfinding to navigate the current procedural maze. It evaluates valid grid cells, avoids walls, and follows calculated routes toward the player or important objective areas.

I also connected the fox’s movement to higher-level game behaviour. Its pressure and aggression could change based on the match state, allowing it to respond differently when it was falling behind rather than behaving identically throughout every round.

The same fox character can be controlled by the AI in single-player mode or by a second person in local multiplayer. The code therefore had to support both control modes while keeping collision, mini-game triggers, rewards, progression, and end conditions consistent.

Development Stage 04

Mini-Games & Polish

The game includes three integrated mini-games: The Sigil’s Echo, Starlight Crossing, and Canopy Cascade. Each mini-game was built as its own system with separate mechanics, controls, timers, win conditions, AI behaviour, reward logic, and transition flow.

The Sigil’s Echo required sequence generation, input validation, progress tracking, timing, and feedback states. Starlight Crossing required a tile-based puzzle with path validation, solvable board generation, rotation controls, countdown logic, and AI solving behaviour. Canopy Cascade introduced arcade-style paddle movement, collision response, scoring, and continuous real-time updates.

A major challenge was connecting each mini-game cleanly to the main state system. A mini-game needed to start from the maze, run independently, determine a result, apply the correct reward or consequence, reset its temporary values, and return to the maze without affecting unrelated systems.

The final polish stage focused on UI spacing, visual feedback, transitions, reward screens, asset loading, countdowns, and consistent screen flow. I also organized the project with Git and GitHub and prepared its code and documentation as a readable software portfolio project.

Troubleshooting

Debugging the Game Systems.

The most difficult bugs came from systems that shaped the entire player experience: state flow, fox behaviour, and procedural maze reliability. Debugging required testing how algorithms, input, AI decisions, collisions, rewards, and screen transitions interacted across complete playthroughs.

Debug File 01

Controlling State Flow Across the Game Loop

Symptom

As the game expanded beyond the main maze, bugs appeared during transitions between the menu, instructions, gameplay, mini-games, reward screens, and end states. A feature could work independently but behave incorrectly when reconnected to the full loop.

Diagnosis

Each screen and mini-game required its own input handling, update logic, drawing logic, temporary variables, and reset conditions. If one state did not clean up fully before the next began, old timers, results, or flags could affect the next round.

Adjustment

I refined the state flow so each mode had clearer entry, update, exit, and reset behaviour. I tested individual transitions first and then completed full playthroughs to confirm that mini-games returned correctly and that rewards, timers, and end conditions did not carry over.

Debug File 02

Tuning Fox Pathfinding and Adaptive Behaviour

Symptom

The fox needed to feel intelligent without becoming unfair. During testing, it could stall near gameplay zones, choose awkward routes, become too passive, or apply too much pressure when the player approached the portal.

Diagnosis

Fox behaviour depended on the live maze grid, BFS route calculation, wall cells, player position, objective areas, portal state, and aggression level. Because the maze changed every round, the AI could not depend on fixed routes.

Adjustment

I tested the fox across different generated layouts and adjusted how it selected targets, followed valid paths, and reacted to the match state. I also tuned its aggression so it could become more threatening without removing the player’s ability to make strategic choices.

Debug File 03

Validating Procedural Mazes and Route Variety

Symptom

The maze could not simply generate random-looking walls. It needed to produce playable and varied layouts where the player and fox were not always forced through one predictable route.

Diagnosis

Randomized depth-first search produced a connected base maze but could still create layouts that felt too linear. Portal placement and alternate paths also needed to remain reachable after each new maze was generated.

Adjustment

I tested repeated generations and refined the logic to add alternate loop openings, clear space around the portal, select distant reachable portal cells, and use a fallback reachable location when an ideal candidate could not be found.

Final Takeaway

Final Reflection.

What This Project Taught Me

The Shifting Grove helped me move from smaller programming exercises to a larger software project with many connected systems. I had to think beyond individual features and consider how maze generation, player movement, fox AI, mini-games, rewards, UI screens, and state transitions worked together inside one game loop.

This strengthened my understanding of object-oriented programming, modular architecture, procedural generation, BFS pathfinding, collision systems, and state management in a practical setting.

The project also taught me how important patience and systematic debugging are in software development. Many issues did not come from one obvious error but from the way several systems interacted. I learned to isolate problems, test one feature at a time, trace how data moved between states, and retest the complete game after a local fix.

Turning software concepts into a playable game made the learning process more engaging while still requiring technical discipline. It helped me become more confident managing a larger codebase, making architectural decisions, using Git for version control, and continuing to polish a project after its main features were functional.

Future Improvements
