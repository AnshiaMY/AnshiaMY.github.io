The Shifting Grove

A Python/Pygame fantasy maze game where players navigate a shifting enchanted grove, complete interactive mini-games, and outsmart an AI-controlled fox guardian through procedurally generated levels and reward-based progression.

Project Overview

Building A World Through Code

The Shifting Grove began as a passion project inspired by my interest in using software to create interactive, imaginative experiences. I wanted to challenge myself beyond smaller programming exercises by building a complete game with structured logic, responsive gameplay, and a polished user experience.

The final concept is a fantasy maze game where players explore an enchanted grove, collect Grove Shards, complete mini-game challenges, and race to escape before the grove shifts. Through this project, I was able to turn algorithms and software concepts into a playable experience while strengthening my programming skills in a fun, creative, and hands-on way.

Development Stage 01

Game Architecture

The game was built using a modular, object-oriented architecture so that each major system could be developed, tested, and debugged independently. Instead of placing all logic inside one large script, I separated the project into focused files and classes for the player, fox, maze generation, shards, UI, asset loading, mini-games, settings, and the main game controller. This made the codebase easier to understand as the project grew and allowed each feature to connect through a clearer structure.

A major part of the architecture was the use of finite state machines and explicit state management. The game has several different modes, including the start menu, instructions, maze gameplay, mini-game transitions, reward screens, win/loss states, and return-to-maze logic. Each state needed defined entry conditions, update behaviour, input handling, and exit transitions so that one screen or mini-game would not interfere with another.

This structure became especially important once the project expanded from a single maze into a full game with three integrated mini-games. The architecture had to support switching between systems, preserving progress, applying rewards, resetting temporary mini-game variables, and returning to the correct maze state. Building this flow helped me practice larger-scale software organization, not just isolated feature development.

Development Stage 02

Maze & Player Systems

The main gameplay is built around procedural maze generation, using a randomized depth-first search approach to create a new navigable layout each round. The maze is generated from a grid structure where paths are carved between cells, producing a playable maze instead of a manually designed map. This allowed the game to feel different across rounds while still maintaining valid routes for the player and fox.

After generation, the maze data had to connect to the rest of the game systems. Wall tiles, walkable paths, collision boundaries, shard locations, mini-game trigger areas, and the final portal objective all depended on the generated layout. This meant the maze was not only a visual background; it acted as the foundation for movement, pathfinding, collision detection, objective placement, and gameplay progression.

The player system handles movement, collision detection, collectible logic, and game-state updates. As the player moves through the grove, the game checks wall collisions, tracks collected Grove Shards, determines when the portal can activate, and handles transitions into mini-games. This required the movement system to work closely with the maze grid and the overall state machine so that exploration, collection, and transitions remained consistent.

Development Stage 03

Fox AI & Game Logic

The fox AI was the core technical challenge of the project. Instead of moving randomly or following a simple scripted path, the fox uses game AI logic and BFS pathfinding to navigate the procedurally generated maze. Because the maze changes each round, the fox cannot rely on a fixed route; it has to reason through the current grid layout and find valid paths based on the live game state.

Breadth-first search was used because it works well for grid-based navigation and can find the shortest path through an unweighted maze. The fox AI uses the maze structure to evaluate neighbouring cells, avoid walls, and move toward relevant targets. This made the fox feel more intentional and allowed it to pressure the player in a way that was connected to the actual maze layout instead of just moving visually across the screen.

The AI also had to interact with more than just movement. The fox behaviour connects to collision logic, player progress, mini-game triggers, reward effects, win conditions, and the Grove Shift system. This made the fox one of the most important integration points in the project because its decisions affected pacing, difficulty, and how the player experienced the game.

The same fox system also supports local multiplayer, where a second player can control the fox instead of the AI. This meant the code had to support both AI-driven and player-driven behaviour without breaking the rest of the game loop. Building this system helped me understand how AI logic, input handling, pathfinding, and state management can be designed to work together inside one flexible game system.

Development Stage 04

Mini-Games & Polish

The game includes three integrated mini-games: The Sigil’s Echo, Starlight Crossing, and Canopy Cascade. Each mini-game was built as its own system with separate mechanics, controls, timers, win conditions, AI behaviour, reward logic, and transition flow. The challenge was not only making each mini-game work on its own, but making each one connect cleanly back to the main maze.

The Sigil’s Echo required sequence logic, input validation, progress tracking, timing, and feedback states. Starlight Crossing required a tile-based puzzle system with path validation, solvable board generation, rotation controls, countdown logic, and AI solving behaviour. Canopy Cascade added arcade-style collision, paddle movement, scoring, and real-time reaction mechanics. Together, the mini-games allowed me to practice different types of gameplay logic within one larger project.

A major part of this stage was debugging integration issues between the mini-games and the main state machine. Each mini-game needed to start from the maze, run independently, determine a result, apply the correct reward or consequence, and return to the maze without carrying over unwanted temporary variables. This required careful handling of resets, timers, player/fox states, and screen transitions.

The final polish stage focused on improving the user experience through UI spacing, visual feedback, transitions, reward screens, image loading, countdowns, and consistent screen flow. I also used Git and GitHub to organize the finished project and prepare it as a readable software portfolio piece. This stage helped turn the project from a functional prototype into a complete, presentable game.

Troubleshooting

Debugging the game systems.

The biggest challenge was not making one feature work once; it was making many systems work together reliably. Each stage of the project introduced integration issues that had to be tested, isolated, and adjusted.

Debug File 01

Managing transitions between game states

Symptom

As more screens and mini-games were added, it became easier for the game to enter the wrong state, skip a transition, or return to the maze incorrectly.

Diagnosis

The issue came from the growing complexity of the game loop. The start menu, instructions, maze, mini-games, rewards, and endings each needed clear rules for when they became active and when they handed control back to another system.

Adjustment

I organized the game around clearer state transitions and tested each path through the game separately. This made it easier to confirm that each screen entered, updated, and exited in the correct order.

Debug File 02

Balancing fox AI behaviour

Symptom

The fox AI needed to feel challenging without making the game frustrating. If the AI behaved too passively, the maze felt empty; if it was too aggressive, the player had less room to explore.

Diagnosis

The fox was not just an enemy sprite. It affected pacing, pressure, mini-game triggers, and the overall feel of the game, so its behaviour had to be tuned within the full gameplay context.

Adjustment

I tested the fox behaviour repeatedly and adjusted how it interacted with the maze and mini-games. This helped make the AI feel more intentional while still leaving the player enough time to make progress.

Debug File 03

Integrating assets, UI, and polish

Symptom

Even after the core mechanics worked, the game did not feel complete until the UI, transitions, backgrounds, reward screens, and asset placement were consistent.

Diagnosis

A playable game can still feel unfinished if its visual feedback and screen flow are not clear. Small issues such as spacing, timing, fonts, image sizing, and transition wording affected how polished the project felt.

Adjustment

I revised the game through repeated polish passes, improving transitions, screen layouts, reward timing, image integration, and UI consistency. This helped the project feel more like a complete portfolio piece rather than only a coding demo.

Final Takeaway

Final Reflection.

What this project taught me.

This project taught me how important organization becomes as a software project grows. At the beginning, individual features felt manageable on their own, but as I added maze generation, player movement, fox behaviour, mini-games, reward screens, and transitions, I had to think more carefully about how each system connected to the rest of the game.

The biggest lesson I took from The Shifting Grove was that polish is part of the engineering process, not something separate from it. Debugging the game meant more than fixing errors; it meant improving flow, feedback, timing, and consistency so the project felt complete. This made me more confident working through a larger software build and showed me the value of iteration, testing, and finishing details.
