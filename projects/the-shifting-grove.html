The Shifting Grove

A Python and Pygame fantasy maze game where players explore a procedurally generated enchanted grove, collect shards, complete three integrated mini-games, and outsmart an AI-controlled fox guardian through pathfinding and reward-based progression.

Project Overview

Building a World Through Code.

The Shifting Grove began as a personal software project inspired by my interest in creating interactive and imaginative experiences. I wanted to move beyond smaller programming exercises by building a complete game with structured logic, responsive gameplay, and a polished user experience.

The project allowed me to turn algorithms and software concepts into a playable experience while strengthening my programming, debugging, and software-architecture skills.

Development Stage 01

Game Architecture

The game was built using a modular, object-oriented architecture so that each major system could be developed, tested, and debugged independently. Instead of placing all logic inside one large script, I separated the project into focused files and classes for the player, fox, maze generation, shards, UI, asset loading, mini-games, settings, and the main game controller. This made the codebase easier to understand as the project grew and allowed each feature to connect through a clearer structure.

A major part of the architecture was the use of finite state machines and explicit state management. The game has several different modes, including the start menu, instructions, maze gameplay, mini-game transitions, reward screens, win/loss states, and return-to-maze logic. Each state needed defined entry conditions, update behaviour, input handling, and exit transitions so that one screen or mini-game would not interfere with another.

This structure became especially important once the project expanded from a single maze into a full game with three integrated mini-games. The architecture had to support switching between systems, preserving progress, applying rewards, resetting temporary mini-game variables, and returning to the correct maze state. Building this flow helped me practice larger-scale software organization, not just isolated feature development.

Development Stage 02

Maze & Player Systems

The main gameplay is built around procedural maze generation, using a randomized depth-first search approach to create a new navigable layout each round. The maze is generated from a grid structure where paths are carved between cells, producing a playable maze instead of a manually designed map. This allowed the game to feel different across rounds while still maintaining valid routes for the player and fox.

After generation, the maze data had to connect to the rest of the game systems. Wall tiles, walkable paths, collision boundaries, shard locations, mini-game trigger areas, and the final portal objective all depended on the generated layout. This meant the maze was not only a visual background; it acted as the foundation for movement, pathfinding, collision detection, objective placement, and gameplay progression.

The player system handles movement, collision detection, collectible logic, and game-state updates. As the player moves through the grove, the game checks wall collisions, tracks collected Grove Shards, determines when the portal can activate, and handles transitions into mini-games. This required the movement system to work closely with the maze grid and the overall state machine so that exploration, collection, and transitions remained consistent.

Development Stage 03

Fox AI & Game Logic

The fox AI was the main technical focus of the game. Instead of using random movement or a fixed chase pattern, the fox uses BFS pathfinding to navigate the procedurally generated maze. Each round, it evaluates the current grid, avoids wall cells, and selects valid routes toward the player or key objective areas.

I also added adaptive behaviour so the fox could respond to the match state. When the fox is falling behind or losing momentum, its behaviour becomes more aggressive, increasing pressure on the player and making the AI feel less predictable. This connected the pathfinding system to higher-level game logic rather than treating movement as a separate feature.

The fox system also interacts with collision detection, player progress, mini-game triggers, reward effects, win conditions, and local multiplayer. Since the same fox character can be controlled by either AI or a second player, the code had to support both modes while keeping the maze, mini-game, and state-transition logic consistent.

Development Stage 04

Mini-Games & Polish

The game includes three integrated mini-games: The Sigil’s Echo, Starlight Crossing, and Canopy Cascade. Each mini-game was built as its own system with separate mechanics, controls, timers, win conditions, AI behaviour, reward logic, and transition flow. The challenge was not only making each mini-game work on its own, but making each one connect cleanly back to the main maze.

The Sigil’s Echo required sequence logic, input validation, progress tracking, timing, and feedback states. Starlight Crossing required a tile-based puzzle system with path validation, solvable board generation, rotation controls, countdown logic, and AI solving behaviour. Canopy Cascade added arcade-style collision, paddle movement, scoring, and real-time reaction mechanics. Together, the mini-games allowed me to practice different types of gameplay logic within one larger project.

A major part of this stage was debugging integration issues between the mini-games and the main state machine. Each mini-game needed to start from the maze, run independently, determine a result, apply the correct reward or consequence, and return to the maze without carrying over unwanted temporary variables. This required careful handling of resets, timers, player/fox states, and screen transitions.

The final polish stage focused on improving the user experience through UI spacing, visual feedback, transitions, reward screens, image loading, countdowns, and consistent screen flow. I also used Git and GitHub to organize the finished project and prepare it as a readable software portfolio piece. This stage helped turn the project from a functional prototype into a complete, presentable game.

Troubleshooting

Debugging the Game Systems.

The most difficult bugs came from systems that shaped the entire player experience: state flow, fox behaviour, and procedural maze reliability. Debugging required testing how algorithms, input, AI decisions, collisions, rewards, and screen transitions interacted across complete playthroughs.

Debug File 01

Controlling State Flow Across the Game Loop

Symptom

As the game expanded beyond the main maze, bugs appeared during transitions between the menu, instructions, gameplay, mini-games, reward screens, and end states. A feature could work independently but behave incorrectly when reconnected to the full loop.

Diagnosis

Each screen and mini-game required its own input handling, update logic, drawing logic, temporary variables, and reset conditions. If one state did not clean up fully before the next began, old timers, results, or flags could affect the next round.

Adjustment

I refined the state flow so each mode had clearer entry, update, exit, and reset behaviour. I tested individual transitions first and then completed full playthroughs to confirm that mini-games returned correctly and that rewards, timers, and end conditions did not carry over.

Debug File 02

Tuning Fox Pathfinding and Adaptive Behaviour

Symptom

The fox needed to feel intelligent without becoming unfair. During testing, it could stall near gameplay zones, choose awkward routes, become too passive, or apply too much pressure when the player approached the portal.

Diagnosis

Fox behaviour depended on the live maze grid, BFS route calculation, wall cells, player position, objective areas, portal state, and aggression level. Because the maze changed every round, the AI could not depend on fixed routes.

Adjustment

I tested the fox across different generated layouts and adjusted how it selected targets, followed valid paths, and reacted to the match state. I also tuned its aggression so it could become more threatening without removing the player’s ability to make strategic choices.

Debug File 03

Validating Procedural Mazes and Route Variety

Symptom

The maze could not simply generate random-looking walls. It needed to produce playable and varied layouts where the player and fox were not always forced through one predictable route.

Diagnosis

Randomized depth-first search produced a connected base maze but could still create layouts that felt too linear. Portal placement and alternate paths also needed to remain reachable after each new maze was generated.

Adjustment

I tested repeated generations and refined the logic to add alternate loop openings, clear space around the portal, select distant reachable portal cells, and use a fallback reachable location when an ideal candidate could not be found.

Final Takeaway

Final Reflection.

What This Project Taught Me

The Shifting Grove helped me grow from writing smaller programming exercises to building a larger software project with many connected systems. I had to think beyond individual features and consider how the maze, player movement, fox AI, mini-games, rewards, UI screens, and state transitions all worked together inside one game loop. This strengthened my understanding of object-oriented programming, modular architecture, procedural generation, BFS pathfinding, collision detection, and finite state machines in a practical setting.

This project also taught me how important debugging, patience, and iteration are in software development. Many issues did not come from one obvious error, but from the way different systems interacted. I learned to slow down, isolate problems, test one feature at a time, and trace how data moved between states. That process made me more confident working through complex bugs instead of feeling stuck when the project became messy.

Personally, this project reminded me that I learn best when I am building something creative and challenging. Turning algorithms and software concepts into a playable game made the learning process more engaging, while still pushing me to improve my technical discipline. It helped me become more comfortable managing a bigger codebase, making design decisions, and continuing to polish a project even after the main features were working.
