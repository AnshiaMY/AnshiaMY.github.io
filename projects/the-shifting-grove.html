The Shifting Grove

A Python/Pygame fantasy maze game built with procedural generation, AI-driven fox behaviour, interactive mini-games, reward-based progression, and custom UI transitions.

Project Overview

From maze idea to playable game.

The Shifting Grove began as a software portfolio project designed to show more than basic programming syntax. I wanted to build a complete game system with movement, collision, procedural level generation, AI behaviour, mini-games, transitions, and a polished user interface.

The final concept became a fantasy maze game where the player explores an enchanted grove, collects Grove Shards, triggers mini-games, and tries to escape before the grove shifts. The game combines a main maze experience with three smaller game modes, creating a larger project that required planning, debugging, and careful state management.

Development Stage 01

Game Architecture

The game was structured around a clear state-based flow so that the start screen, instructions, maze gameplay, mini-games, reward screens, and end states could each behave separately while still connecting into one complete experience. This helped prevent the project from becoming one long file of unrelated logic.

Separate files and classes were used for major systems such as the player, fox, maze, shards, UI, mini-games, settings, and asset loading. This made the project easier to debug because I could isolate problems by system instead of searching through the entire game at once.

Development Stage 02

Maze & Player Systems

The main game is built around a procedurally generated maze, giving each round a different layout instead of relying on one fixed map. The player moves through the maze, collects Grove Shards, avoids the fox guardian, and activates the portal after meeting the required conditions.

Collision detection, wall placement, player movement, shard collection, and portal logic all had to work together so the maze felt playable rather than just visually generated. This part of the project helped me understand how game systems depend on both visual layout and behind-the-scenes rules.

Development Stage 03

Fox AI & Game Logic

The fox guardian was designed as an opposing force in the game rather than just a decorative character. In single-player mode, the fox uses AI behaviour to move through the maze, pressure the player, interact with the game state, and create a sense of challenge.

The game also supports local multiplayer, where a second player can control the fox instead of the AI. This meant the logic had to support both AI-controlled and player-controlled behaviour while keeping the rest of the game systems consistent.

Development Stage 04

Mini-Games & Polish

The game includes three integrated mini-games: The Sigil’s Echo, Starlight Crossing, and Canopy Cascade. Each mini-game has its own rules, controls, visuals, countdowns, win conditions, and reward states, while still connecting back into the larger maze experience.

A major part of the polish involved improving transitions, reward screens, UI spacing, font consistency, visual feedback, and asset loading. These details made the game feel more complete and helped turn it from a working prototype into a stronger software portfolio project.

Troubleshooting

Debugging the game systems.

The biggest challenge was not making one feature work once; it was making many systems work together reliably. Each stage of the project introduced integration issues that had to be tested, isolated, and adjusted.

Debug File 01

Managing transitions between game states

Symptom

As more screens and mini-games were added, it became easier for the game to enter the wrong state, skip a transition, or return to the maze incorrectly.

Diagnosis

The issue came from the growing complexity of the game loop. The start menu, instructions, maze, mini-games, rewards, and endings each needed clear rules for when they became active and when they handed control back to another system.

Adjustment

I organized the game around clearer state transitions and tested each path through the game separately. This made it easier to confirm that each screen entered, updated, and exited in the correct order.

Debug File 02

Balancing fox AI behaviour

Symptom

The fox AI needed to feel challenging without making the game frustrating. If the AI behaved too passively, the maze felt empty; if it was too aggressive, the player had less room to explore.

Diagnosis

The fox was not just an enemy sprite. It affected pacing, pressure, mini-game triggers, and the overall feel of the game, so its behaviour had to be tuned within the full gameplay context.

Adjustment

I tested the fox behaviour repeatedly and adjusted how it interacted with the maze and mini-games. This helped make the AI feel more intentional while still leaving the player enough time to make progress.

Debug File 03

Integrating assets, UI, and polish

Symptom

Even after the core mechanics worked, the game did not feel complete until the UI, transitions, backgrounds, reward screens, and asset placement were consistent.

Diagnosis

A playable game can still feel unfinished if its visual feedback and screen flow are not clear. Small issues such as spacing, timing, fonts, image sizing, and transition wording affected how polished the project felt.

Adjustment

I revised the game through repeated polish passes, improving transitions, screen layouts, reward timing, image integration, and UI consistency. This helped the project feel more like a complete portfolio piece rather than only a coding demo.

Final Takeaway

Final Reflection.

What this project taught me.

This project taught me how important organization becomes as a software project grows. At the beginning, individual features felt manageable on their own, but as I added maze generation, player movement, fox behaviour, mini-games, reward screens, and transitions, I had to think more carefully about how each system connected to the rest of the game.

The biggest lesson I took from The Shifting Grove was that polish is part of the engineering process, not something separate from it. Debugging the game meant more than fixing errors; it meant improving flow, feedback, timing, and consistency so the project felt complete. This made me more confident working through a larger software build and showed me the value of iteration, testing, and finishing details.
