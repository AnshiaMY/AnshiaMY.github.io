VEX Toy-Sorting Claw Machine

A group VEX robotics project focused on designing, building, programming, and testing an autonomous claw-based sorting machine. The robot used touch input, sensor-based navigation, a motorized claw-and-lift mechanism, and modular C++ control logic to transport objects into selected category zones.

Project Overview

From Playroom Mess to Smart Cleanup.

The project began with the idea of turning a repetitive household task into a structured robotics challenge. Toys and small objects often need to be collected and organized after use, so our group designed a VEX robot that could model part of that cleanup process by transporting an object to a selected category area.

The goal was not only to build a moving claw mechanism, but to create an autonomous system with a clear input–action–output sequence. Through simple user input, the robot introduced a colour-to-category association where each selection represented a sorting destination and the robot responded by completing the matching task.

This gave the project both a practical focus and an educational layer: the robot demonstrated automated sorting while reinforcing organization, colour recognition, and cause-and-effect behaviour.

Build Stage 01

Mechanical Frame & Drive Base

The robot’s frame and drive base were designed from the VEX IQ kit components provided for the project rather than from a prebuilt chassis. After the claw and vertical-lift concepts were developed, the base was designed around those mechanisms so the robot could support the VEX Brain, drive motors, lift motor, claw motor, sensors, and object-handling system within one stable structure.

Because the robot needed to carry an object above the drive platform, the frame had to be wide enough to improve stability and reduce tipping during movement. At the same time, the wheel placement and motor positioning had to support straight-line driving, controlled turning, and alignment with the sorting area.

The final frame also needed enough mounting space for the lift mechanism and sensors while remaining compact enough to navigate the sorting path reliably.

Build Stage 02

Claw & Lift Mechanism

The object-handling system combined a two-pronged overhead claw with a vertical-lift mechanism. During the design process, our group considered several options for securing the object, including a front-facing claw, a pronged lift, and an overhead hook.

After testing an early front-facing claw concept, the design was changed because the object grip was not consistent enough. The overhead claw was selected instead because it allowed the user to place the object more directly into the mechanism, improving reliability during the sorting sequence.

To raise and lower the claw, our group developed a chain-based vertical lift inspired by forklift-style lifting systems. Since the VEX kit did not use hydraulics, the lift was adapted using a motor-driven sprocket, a second sprocket, and a closed-loop chain.

The lift motor turned the top sprocket, moving the chain and allowing the claw assembly to travel vertically. This gave the robot a controlled way to raise the object for transport and lower or release it near the selected category box.

Integrating the claw with the lift also required mechanical trade-offs. Because the chain carried the weight of both the claw and the object, the claw assembly had to be kept as light as possible, and additional supports were added to reduce stress on the plastic VEX components.

Build Stage 03

Sensor-Based Navigation

The robot used multiple sensors to connect user input with autonomous movement. At the beginning of the sequence, the touch sensor counted one, two, or three taps and converted that input into a selected sorting category.

Once the category was chosen, the robot used its colour sensor to scan the floor for the matching coloured stripe, which acted as a path marker for the correct drop-off area.

Additional sensors helped make the movement more controlled after the correct stripe was detected. The gyro sensor supported the 90-degree turn toward the category box, while the distance sensor helped the robot stop at a consistent distance from the box wall before releasing the object.

Motor-encoder feedback was also used to track movement so the robot could return more reliably toward its starting position. Together, these sensors allowed the robot to make decisions based on feedback rather than relying only on fixed timing.

Build Stage 04

Autonomous Sorting Sequence

The robot’s autonomous behaviour was programmed in C++ using VEXcode and structured around the full sorting cycle rather than one long timed routine.

I divided the code into separate functions for the main stages of the sequence: reading touch-sensor input, controlling the claw and lift, navigating toward the selected stripe, aligning with the drop-off box, releasing the object, and returning to the starting position.

This modular structure made the program easier to test because each subsystem could be debugged individually before being connected into the full autonomous routine.

The code also included recovery logic for two common issues that could occur during testing. If the robot missed the expected colour stripe, encoder-based movement limits prevented it from continuing forward indefinitely and provided a backup condition for ending the search step.

If the robot approached the box at a slightly different position than expected, the distance sensor helped correct the stopping point before the drop-off sequence began. These safeguards made the routine more reliable because the program did not depend only on perfect sensor readings or perfectly repeatable physical movement.

The code also had to account for real-world movement errors. Gyro-based turns were not always exactly 90 degrees, and sensor readings could vary depending on alignment, surface reflection, or distance from the target.

To improve consistency, the program used calibrated thresholds, encoder-based movement tracking, and repeated testing to tune how the robot moved between each stage. The main coding challenge was coordinating sensor readings, motor commands, and mechanical timing so the physical robot could complete the same sorting sequence reliably across multiple trials.

Troubleshooting

Debugging the Sorting Sequence.

Testing showed that the robot’s reliability depended on how effectively its mechanical design, sensors, and C++ control logic worked together. Several parts of the design had to be revised as the team moved from early concepts to a working demonstration robot.

Debug File 01

Claw Redesign and Grip Reliability

Symptom

The early object-handling concepts were not reliable enough for a repeatable sorting sequence. A front-facing claw was tested, but its ability to grasp the objects consistently depended too much on object placement and alignment.

Diagnosis

The issue was mechanical rather than only software-related. If the object did not sit properly in the claw, the robot could not carry it securely through the drive, turn, and drop-off stages. This made the full autonomous routine less dependable, even if the code sequence was correct.

Adjustment

The group moved away from the front-facing claw and redesigned the mechanism as a two-pronged overhead claw. This made it easier for the user to place the object directly into the mechanism and improved the robot’s ability to hold the object during transport and release it during the final drop-off.

Debug File 02

Lift Positioning and Mechanical Constraints

Symptom

The lift mechanism needed to raise the claw high enough to transport and release the object, but the robot also had to remain stable and avoid placing too much stress on the plastic VEX components.

Diagnosis

The chain-based vertical lift introduced several design constraints. The claw and object added weight to the chain, while the redesigned chassis created a central gap for the lift mechanism.

This made it difficult to use a bumper sensor to detect the lowest claw position. The available lift height also limited how easily objects could be dropped over taller box walls.

Adjustment

Additional supports were added to reduce stress on the chain system, and the claw assembly was simplified so it used only the necessary components.

Since the bumper sensor was no longer practical, the software relied on motor-encoder thresholds to estimate lift position and control the mechanism more reliably.

Debug File 03

Sensor Calibration and Navigation Accuracy

Symptom

The robot did not always move with perfect accuracy during navigation. Distance readings sometimes varied, the colour sensor could be affected by lighting or surface reflections, and gyro-based turns did not always stop at exactly 90 degrees.

Diagnosis

The autonomous routine depended on several sensors working together. The colour sensor had to detect the correct floor stripe, the gyro had to support the turn toward the box, the distance sensor had to stop the robot near the wall, and motor encoders had to support return movement.

When one reading was slightly inconsistent, the robot’s final alignment could shift.

Adjustment

The group improved reliability through repeated calibration and software tuning. Controlled stripe colours and distances were used for colour detection, tolerance margins were added for distance sensing, and gyro-based turns were adjusted through multiple trials.

Demo-day travel distances were also shortened so the robot could complete a more consistent and repeatable sorting run.

Final Takeaway

Final Reflection.

What This Project Taught Me

This project taught me that engineering is often about working within constraints rather than simply building every idea that first comes to mind. At the start, there were many features and design ideas we wanted to include, but limitations such as time, available components, and project requirements meant we had to plan carefully and prioritize what was realistic.

This showed me the importance of designing first and thinking ahead because real engineering projects always include specifications, restrictions, and trade-offs that shape the final solution.

Building a robot capable of completing a full sequence of tasks also showed me that success is not only about making individual parts work. The mechanical structure, sensors, motors, and program logic must work together reliably as one complete system.

Through testing and refining the design, I improved my ability to adapt when ideas did not work as expected and to make practical decisions based on evidence rather than assumptions.

Working as part of a team was another valuable part of the project. It required clear communication, shared problem-solving, and coordination between members responsible for different areas of the robot.

We regularly discussed design decisions, test results, and changes so everyone’s work remained aligned. My main contributions focused on software, sensor integration, program logic, documentation, and presentation work.

Collaborating closely with the mechanical side of the project helped me better understand how software decisions depend on physical design constraints.

Future Improvements

Next Iteration Notes

  • Improve the lift mechanism so the claw can raise objects high enough for more reliable drop-off into enclosed boxes.
  • Strengthen the claw-and-lift assembly to reduce stress on the VEX components during repeated sorting cycles.
  • Improve sensor calibration for distance sensing, colour-stripe detection, and gyro-based turning accuracy.
  • Refine the software logic with additional error handling so the robot can recover more reliably if a stripe is missed or a sensor reading is inconsistent.
  • Test the full sequence with more object shapes, weights, and category layouts to evaluate repeatability beyond the demonstration setup.
  • Improve team documentation during testing so design changes, sensor values, and code adjustments are recorded more consistently between trials.
