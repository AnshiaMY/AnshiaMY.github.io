VEX Toy-Sorting Claw Machine

A group VEX robotics project focused on designing, building, and testing an autonomous claw-based sorting machine. The robot used a claw-and-lift mechanism, sensor-based navigation, and C++ control logic to move objects into selected category zones.

Project Overview

From Playroom Mess To Smart Cleanup.

The project began with the idea of turning a repetitive household task into a structured robotics challenge. Toys and small objects often need to be collected and organized after use, so our group designed a VEX robot that could model part of that cleanup process by transporting an object to a selected category area. The goal was not only to build a moving claw mechanism, but to create an autonomous system with a clear input–action–output sequence. Through simple user input, the robot introduced a colour-to-category association, where each selection represented a sorting destination and the robot responded by completing the matching task. This gave the project both a practical focus and an educational layer: the robot demonstrated automated sorting while reinforcing organization, colour recognition, and cause-and-effect behaviour.

Build Stage 01

Mechanical Frame & Drive Base

The robot’s frame and drive base were designed from the VEX IQ kit components provided for the project, rather than from a prebuilt chassis. After the claw and vertical lift concepts were developed, the base was designed around those mechanisms so the robot could support the VEX Brain, drive motors, lift motor, claw motor, sensors, and object-handling system within one stable structure.

Because the robot needed to carry an object above the drive platform, the frame had to be wide enough to improve stability and reduce tipping during movement. At the same time, the wheel placement and motor positioning had to support straight-line driving, controlled turning, and alignment with the sorting area. The final frame also needed enough mounting space for the lift mechanism and sensors while remaining compact enough to navigate the sorting path reliably.

Build Stage 02

Claw & Lift Mechanism

The object-handling system combined a two-pronged overhead claw with a vertical lift mechanism. During the design process, our group considered several options for securing the object, including a front-facing claw, a pronged lift, and an overhead hook. After testing an early front-facing claw concept, the design was changed because the object grip was not consistent enough. The overhead claw was selected instead because it allowed the user to place the object more directly into the mechanism, which improved reliability during the sorting sequence.

To raise and lower the claw, our group developed a chain-based vertical lift inspired by forklift-style lifting systems. Since the VEX kit did not use hydraulics, the lift was adapted using a motor-driven sprocket, a second sprocket, and a closed-loop chain. The lift motor turned the top sprocket, which moved the chain and allowed the claw assembly to travel vertically. This gave the robot a controlled way to raise the object for transport and lower or release it near the selected category box.

Integrating the claw with the lift also required mechanical trade-offs. Because the chain carried the weight of both the claw and the object, the claw assembly had to be kept as light as possible, and additional supports were added to reduce stress on the plastic VEX components. This made the final mechanism more reliable while still allowing the claw and lift to work together as one object-handling subsystem.

Build Stage 03

Sensor-Based Navigation

The robot used multiple sensors to connect user input with autonomous movement. At the beginning of the sequence, the touch sensor counted one, two, or three taps and converted that input into a selected sorting category. Once the category was chosen, the robot used its colour sensor to scan the floor for the matching coloured stripe, which acted as a path marker for the correct drop-off area. Additional sensors helped make the movement more controlled after the correct stripe was detected. The gyro sensor supported the 90-degree turn toward the category box, while the distance sensor helped the robot stop at a consistent distance from the box wall before releasing the object. Motor encoder feedback was also used to track movement so the robot could return more reliably toward its starting position. Together, these sensors allowed the robot to make decisions based on feedback rather than relying only on fixed timing.

Build Stage 04

Autonomous Sorting Sequence

The robot’s autonomous behaviour was programmed in C++ using VEXcode and structured around the full sorting cycle rather than a single timed routine. I divided the code into separate functions for the main stages of the sequence: reading the touch sensor input, controlling the claw and lift, navigating toward the selected stripe, aligning with the drop-off box, releasing the object, and returning to the start. This modular structure made the program easier to test because each subsystem could be debugged individually before being connected into the full autonomous routine.

The code also included recovery logic for two common issues that could happen during testing. If the robot missed the expected colour stripe, encoder-based movement limits prevented it from continuing forward indefinitely and gave the program a backup condition for ending the search step. If the robot approached the box at a slightly different position than expected, the distance sensor helped correct the stopping point before the drop-off sequence began. These safeguards made the routine more reliable because the program did not depend only on perfect sensor readings or perfectly repeatable physical movement.

The code also had to account for real-world movement errors. Gyro-based turns were not always exactly 90 degrees, and sensor readings could vary depending on alignment, surface reflection, or distance from the target. To improve consistency, the program used calibrated thresholds, encoder-based movement tracking, and repeated testing to tune how the robot moved between each stage. The main coding challenge was coordinating sensor readings, motor commands, and mechanical timing so the physical robot could complete the same sorting sequence reliably across multiple trials.

Troubleshooting

Debugging The Game Systems.

The most difficult bugs came from the systems that shaped the entire player experience: game-state flow, fox AI behaviour, and procedural maze reliability. Debugging the game required more than fixing isolated errors; I had to test how algorithms, player input, AI decisions, collision logic, and screen transitions interacted across repeated playthroughs.

Debug File 01

Controlling state flow across the full game loop

Symptom

As the game expanded beyond the main maze, bugs started appearing during transitions between the menu, instructions, gameplay, mini-games, reward screens, and win/loss states. A feature could work on its own but behave incorrectly when connected back into the full game loop.

Diagnosis

The issue was caused by overlapping state responsibilities. Each screen and mini-game needed its own input handling, update logic, drawing logic, temporary variables, and reset conditions. If one state did not fully clean up before the next one started, old values could affect the next round.

Adjustment

I refined the state-management flow so each mode had clearer entry, update, exit, and reset behaviour. I tested transitions individually, then tested full playthroughs to make sure mini-games returned to the maze correctly and that rewards, timers, and end conditions did not carry over unintentionally.

Debug File 02

Tuning fox AI, BFS pathfinding, and adaptive behaviour

Symptom

The fox needed to feel intelligent without becoming unfair. During testing, I had to debug cases where the fox stalled near important gameplay zones, chose awkward paths, became too passive, or applied too much pressure when the player was trying to reach the portal.

Diagnosis

The fox behaviour depended on several connected systems: BFS pathfinding, the current maze grid, wall collisions, player position, objective areas, portal state, and the fox’s aggression level. Because the maze changed each round, the AI had to respond to the live layout rather than relying on fixed routes.

Adjustment

I tested the fox across different maze layouts and adjusted how it selected targets, followed valid paths, and reacted to the match state. I also tuned its adaptive aggression so it could become more threatening when falling behind while still leaving the player enough room to make strategic choices.

Debug File 03

Validating procedural maze generation and route variety

Symptom

The maze could not just generate random-looking walls. It needed to create layouts that were playable, winnable, and varied enough that the player and fox were not forced into one predictable route every round.

Diagnosis

The base maze was generated using randomized depth-first search, which creates a connected maze but can still feel too linear. To improve gameplay, I added extra loop openings, checked far reachable portal locations, and verified that the portal had multiple possible paths from the starting area.

Adjustment

I tested repeated maze generations and refined the generation logic so the game could carve a connected base maze, add alternate routes, clear space around the portal, and fall back to a far reachable tile if an ideal portal location was not found. This made the maze more reliable, replayable, and fair for both the player and fox AI.

Final Takeaway

Final reflection

What this project taught me.

This project taught me that engineering is often about working within constraints rather than simply building every idea you first imagine. At the start, there were many features and design ideas we wanted to include, but limitations such as time, available components, and the project requirements meant we had to carefully plan and prioritise what was realistic. This showed me the importance of designing first and thinking ahead, because in real engineering there are always specifications, restrictions, and trade-offs that shape the final solution. Building a robot capable of completing a full sequence of tasks also showed me that success is not just about making individual parts work, but making sure they work together reliably as one complete system. Through testing and refining our design, I improved my ability to adapt when ideas did not work as expected and to make practical decisions based on evidence rather than assumptions.

Working as part of a team was one of the most valuable parts of this project, as it required clear communication, shared problem-solving, and coordination between members responsible for different parts of the robot. We had to regularly discuss design decisions, test results, and changes so that everyone’s work stayed aligned and the robot functioned as one complete system. My main contributions were focused on software, sensor integration, program logic, documentation, and presentation work, and collaborating closely with the mechanical side of the project helped me better understand how software decisions depend on physical design constraints.

Future Improvements

Next iteration notes

  • Expand the fox AI with stronger decision-making so it can switch between chasing the player, guarding the portal, blocking key routes, and responding to mini-game opportunities.
  • Refine the maze-generation system with more controlled difficulty scaling, route variety, and layout rules so each generated maze feels both random and intentionally designed.
  • Improve code scalability by reducing repeated logic, cleaning up shared systems, and making the project easier to expand with new levels, mechanics, or additional mini-games.
  • Continue playtesting to better balance fox aggression, mini-game timers, reward effects, and win/loss conditions across both single-player and local multiplayer modes.
  • Explore deployment options for a web-playable version so the game can eventually be experienced directly from the portfolio.
