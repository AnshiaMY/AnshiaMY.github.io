Autonomous Line-Following Robot

A Raspberry Pi–based mobile robot designed to follow a black line using infrared sensor feedback, PID control, and iterative testing, with additional containment testing using a black circular boundary.

Project Overview

From Concept to Control.

This project began as an exploration of feedback-based movement. Instead of programming the robot to follow a fixed route, the goal was to use sensor data to guide its decisions: tracking a black line during path-following tests and detecting the edge of a black circular boundary during containment testing.

Design Note 01

CAD & Chassis Design

The chassis was designed in SolidWorks and 3D printed as the mechanical foundation of the robot. The design had to account for component placement, sensor height, motor alignment, wiring space, and overall stability while keeping the structure compact enough for smooth movement.

The final layout positioned the infrared sensors close to the surface for reliable contrast detection and provided secure mounting points for the Raspberry Pi, motors, motor driver, custom circuit board, and supporting electronics.

Fabrication Note 02

Circuit Board Fabrication

The circuit board was fabricated from a plain copper board using a custom digital schematic and board layout. After the design was heat-transferred onto the copper surface, the board was etched to form the electrical traces used to connect the robot’s sensors, motor driver, power connections, and control electronics.

Assembly Note 03

Wiring & Assembly

Once the board was prepared, the components were soldered and wired into the robot. The infrared sensors, motor driver, DC motors, and regulated 5V/3.3V power connections had to work together reliably, so wiring checks and multimeter-based debugging were used before moving into full system testing.

Control Note 04

Control Logic & Testing

The Raspberry Pi ran Python code that continuously read the two bottom-mounted infrared sensors and compared their black-and-white contrast states to determine the robot’s position relative to the line.

Based on the left and right sensor readings, the program adjusted the two DC motor outputs to correct the robot’s path in real time. A PID controller was tuned across six testing iterations to smooth the motor response, reduce oscillation, and improve stability during straight-line tracking and racetrack turns.

Testing the Robot

On the Track.

The robot was tested across three track conditions, each focused on a different behaviour: straight-line tracking, curved-path correction, and boundary detection. Together, the tests showed how the same sensing and motion systems responded to different control challenges.

01

Straight Line Test

The racetrack path introduced a more complex line-following condition by requiring the robot to respond to continuous curves and direction changes. This test helped evaluate how smoothly the control logic adjusted the motor outputs during turns, while also revealing moments of overcorrection as the sensors reacted to changes in the path.

  • Verified black-and-white contrast detection
  • Observed real-time motor correction
  • Established a baseline before curved and boundary-based testing
02

Racetrack Path Test

The racetrack introduced continuous curves and direction changes. This test evaluated how smoothly the PID controller adjusted the motor outputs during turns while also revealing moments where aggressive corrections caused the robot to oscillate or oversteer.

  • Tested curved-path tracking
  • Evaluated turning response
  • Identified overcorrection during direction changes
03

Black Circle Boundary Test

This test was inspired by sumo robot competitions, where an autonomous robot must remain inside a marked arena. The robot was placed inside a black circular boundary to explore whether the same infrared sensors used for line tracking could support a different behaviour.

Instead of following a continuous line, the robot had to recognize when it approached the marked edge and adjust its movement to remain inside the boundary.

  • Tested circular boundary detection
  • Applied the sensor system to a containment challenge
  • Demonstrated multiple uses for the same sensing hardware

Troubleshooting

Debugging the Build.

Testing revealed that the robot’s behaviour depended on how well the mechanical design, sensors, wiring, and control logic worked together. Each issue had to be observed, diagnosed, and adjusted before the complete system became more reliable.

Debug File 01

Sensor Height vs. Tracking Accuracy

Symptom

The robot’s tracking behaviour changed when the sensor position shifted slightly or when the robot moved across uneven parts of the test surface.

Diagnosis

The issue was not only in the code. The infrared sensors depended on their height and angle relative to the surface, so small mechanical placement changes affected both the sensor readings and control response.

Adjustment

I treated the problem as both a mechanical and software issue by checking sensor placement, testing across different track conditions, and adjusting the control response to better match the observed sensor behaviour.

Debug File 02

Motor Overcorrection During Turns

Symptom

During curved-path testing, the robot sometimes corrected too sharply when entering or leaving a turn.

Diagnosis

The motor response was too aggressive when one infrared sensor detected the line before the other, causing the robot to oversteer instead of smoothly returning toward the path.

Adjustment

The PID values were tuned through repeated trials to smooth the correction, reduce oscillation, and improve stability during curved-path tracking.

Debug File 03

Wiring & System Integration

Symptom

Small connection issues affected the robot’s overall behaviour, especially when the sensors, motors, power system, and control hardware were tested together.

Diagnosis

The robot combined a custom-fabricated circuit board, soldered components, regulated power connections, infrared sensors, DC motors, a motor driver, and Raspberry Pi control. One weak connection could therefore affect several parts of the system.

Adjustment

I tested the system incrementally, inspected wiring and solder joints, verified power connections, and confirmed each subsystem before running the complete robot.

Final Takeaway

Final Reflection.

What This Project Taught Me

This project taught me that debugging a physical robot requires both technical problem-solving and persistence. The robot did not become reliable after one round of coding or testing; it improved through repeated observation, small adjustments, and retesting.

This project taught me that debugging a physical robot requires both technical problem-solving and persistence. The robot did not become reliable after one round of coding or testing; it improved through repeated observation, small adjustments, and retesting. When the robot drifted, overcorrected, or behaved inconsistently, the cause was not always obvious because the issue could come from sensor placement, motor response, wiring reliability, or the control logic. I learned to slow down, isolate one variable at a time, and treat each test result as evidence for the next improvement.

The biggest lesson I took from this project was that failed tests are an important part of engineering, not just signs that something went wrong. Each issue helped me understand the system more clearly and pushed me to become more patient, resilient, and systematic in how I troubleshoot. This project made me more confident working through uncertainty and showed me that strong engineering comes from continuing to test, debug, and refine a design even when the solution is not immediate.

Future Improvements

Next Iteration Notes

  • Improve the physical organization of the robot with cleaner wiring and stronger cable management.
  • Use more secure sensor mounting to maintain consistent height and angle during testing.
  • Continue tuning the PID controller for smoother turns and more stable line-following performance.
  • Record consistent performance data across additional track layouts to compare tuning iterations more clearly.
