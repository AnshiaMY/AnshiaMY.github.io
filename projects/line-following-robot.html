Autonomous Line-Following Robot

A Raspberry Pi–based mobile robot designed to follow a black line using sensor feedback, PID control, and iterative testing, with additional boundary-detection testing using a black-circle condition.

Raspberry Pi Python SolidWorks Motor Control
View Demos Final Reflection

Project Overview

From concept to control.

This project began as an exploration of feedback-based movement. Instead of programming the robot to follow a fixed route, the goal was to use sensor data to guide its decisions: tracking a black line during path-following tests and detecting the edge of a black-circle boundary during containment testing.

01

CAD & Chassis Design

The chassis was designed in SolidWorks and 3D printed as the mechanical foundation of the robot. The design had to account for component placement, sensor height, motor alignment, wiring space, and overall stability, while keeping the structure compact enough for smooth movement. The final layout positioned the infrared sensors close to the surface for reliable contrast detection and provided secure mounting points for the Raspberry Pi, motors, and supporting electronics.

02

Circuit Board Fabrication

The circuit board was fabricated from a plain copper board using a custom digital schematic and board layout. After the design was heat-transferred onto the copper surface, the board was etched to form the electrical traces used to connect the robot’s sensors, motor driver, power connections, and control electronics.

03

Wiring & Assembly

Once the board was prepared, the components were soldered and wired into the robot. The infrared sensors, motor driver, DC motors, and regulated 5V/3.3V power connections had to work together reliably, so wiring checks and multimeter-based debugging were used before moving into full system testing.

04

Control Logic & Testing

The Raspberry Pi ran Python code that continuously read the two bottom-mounted infrared sensors and compared their black-and-white contrast readings to determine the robot’s position relative to the line. Based on the left and right sensor states, the program adjusted the DC motor outputs to correct the robot’s path in real time. A PID controller was tuned through repeated trials to smooth the motor response, reduce oscillation, and improve stability during straight-line tracking, racetrack turns, and boundary-detection testing.

Testing the Robot

On The Track.

The robot was tested across three track conditions, each focused on a different behaviour: straight-line tracking, curved-path correction, and boundary detection. Below is the final robot and its three test variations.

01

Straight Line Test

This test used a mostly straight black path with a small deviation to establish a baseline for the robot’s tracking behaviour. By starting with a simple path before moving to more complex layouts, I could observe whether the infrared sensors were detecting contrast consistently and whether the motor corrections were steady enough to keep the robot aligned.

  • Verified black-and-white contrast detection
  • Observed real-time motor correction
  • Established a baseline before curved and boundary-based testing
02

Racetrack Path Test

The racetrack path introduced a more complex line-following condition by requiring the robot to respond to continuous curves and direction changes. This test helped evaluate how smoothly the control logic adjusted the motor outputs during turns, while also revealing moments of overcorrection as the sensors reacted to changes in the path.

  • Tested curved-path tracking
  • Evaluated turning response
  • Identified overcorrection during direction changes
03

Black Circle Test

The black circle test was inspired by sumo robot competitions, where an autonomous robot must stay inside a circular ring while responding to its surroundings. For this test, the robot was placed inside a dark circular boundary to explore whether the same infrared sensor logic used for line tracking could also support boundary detection. Instead of following a single path, the robot had to recognize when it approached the edge of the marked area and adjust its movement to remain within the boundary.

  • Tested boundary detection
  • Connected line-sensing logic to a sumo-style arena challenge
  • Showed how the same sensors could support a different robot behaviour

Troubleshooting

Debugging The Build.

Testing revealed that the robot’s behaviour depended on how well the mechanical design, sensors, wiring, and control logic worked together. Each issue had to be observed, diagnosed, and adjusted before the system became more reliable.

Sensor Height vs. Tracking Accuracy

Symptom: The robot’s tracking behaviour changed when the sensor position shifted slightly or when the robot moved over uneven parts of the track.

Diagnosis: The issue was not only in the code. The infrared sensors depended on their height and angle relative to the surface, so small mechanical placement changes affected the sensor readings and the control response.

Adjustment: I treated the problem as both a mechanical and software issue by checking sensor placement, testing the robot across different track conditions, and adjusting the control logic so the motor response matched the sensor behaviour more reliably.

Motor Overcorrection During Turns

Symptom: During curved-path testing, the robot sometimes corrected too sharply when entering or leaving a turn.

Diagnosis: The motor response was too aggressive when one infrared sensor detected the line before the other, causing the robot to oversteer instead of smoothly correcting back toward the path.

Adjustment: PID values were tuned through repeated trials to smooth the correction, reduce oscillation, and improve the robot’s stability during curved-path tracking.

Wiring & System Integration

Symptom: Small connection issues affected the robot’s overall behaviour, especially when testing the sensors, motors, and control hardware together.

Diagnosis: The robot combined a custom-fabricated circuit board, soldered components, power connections, infrared sensors, DC motors, and Raspberry Pi control, so one weak connection could affect multiple parts of the system.

Adjustment: I tested the system step by step, checked wiring and soldered joints, verified power connections, and confirmed each subsystem before running the full robot.

Final Takeaway

Final Reflection

This project showed me how closely software, electronics, and mechanical design are connected in a real robotic system. The robot’s performance did not depend on one part working in isolation; it depended on the chassis design, sensor placement, wiring, power connections, and control logic all working together. The most valuable part of the project was testing the robot under different track conditions and seeing how small physical changes affected its behaviour. A slight change in sensor height, motor response, or wiring reliability could change how the robot followed the line, corrected its path, or reacted to a boundary. This helped me understand mechatronics as a full-system design process rather than separate mechanical, electrical, and software tasks.

Future Improvements

  • Improve the physical organization of the robot with cleaner wiring and better cable management.
  • Use more secure sensor mounting to increase reliability during testing.
  • Continue tuning the control system to achieve smoother turning and more stable line-following performance.
  • Collect more consistent testing data across different track layouts to better compare performance between trials.
